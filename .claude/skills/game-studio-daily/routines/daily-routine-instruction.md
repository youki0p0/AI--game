# 毎朝Routine — game-studio-daily（dailyモード）

このファイルはスケジュールRoutineのプロンプトとして使用します。
`/game-studio-daily` スキルの有無にかかわらず、このファイルだけで毎日のゲーム開発Routineが完全に動きます。

---

## 実行指示

対象リポジトリ `https://github.com/youki0p0/AI--game` の現在の状態、Issue、PR、直近の差分を確認し、今日中に「プレイヤーが体感できる小さな遊び」を1つ完成させるための開発方針を日本語で整理してください。

---

## 必ず守ること（安全ルール）

- ログ、Routine報告、QA結果、リファクタ報告、Issue要約、PR要約、意思決定記録は必ず日本語で出力する
- コード、ファイルパス、コマンド名、API名、モデル名、ブランチ名、コミットハッシュは英語のままでよい
- 大きな機能追加ではなく、今日触って分かる小さな変化を優先する
- 他ゲームの参考要素は、そのままコピーせず、面白さの構造だけ抽出する
- 勝手に大規模リファクタリングしない
- 機能追加PRとリファクタリングPRは分ける
- 本番DB変更、本番環境変更、破壊的変更、削除は行わず、必要な場合は提案に留める
- 破壊的変更・強制push・秘密情報の露出は行わない

---

## 実行前に必ず確認するもの

利用可能であれば以下をすべて確認する。確認できない情報は推測せず「確認できませんでした」と明記する。

```
git status
git log --oneline -10
git diff HEAD~1
gh issue list --state open
gh pr list --state open
README.md / CLAUDE.md / package.json
```

## 出力形式（必ずこの順序で日本語で出力する）

# 毎日のゲーム開発ログ

## 1. 現在の状況
- リポジトリの状態（ブランチ・未コミット差分・最新コミット）
- 未完了 Issue（確認できなければその旨を明記）
- 未完了 PR（確認できなければその旨を明記）
- 直近の差分まとめ
- 注意すべき問題

## 2. 今日作るプレイヤー体験
今日中に完成させる、プレイヤーが触って分かる小さな遊びを1つだけ決める。
「〇〇すると〇〇が起きる」という形で1文で書く。

## 3. プレイヤー代表の要望
現実のゲームにある面白い体験を参考に今日の改善案を出す。
表現・名称・UI・数値・文章はコピーしない。面白さの構造だけ抽出する。

## 4. ディレクター判断
- 採用すること
- 保留すること
- 今日やらないこと

## 5. 今日の最小仕様
ゲームプランナーとして、今日実装できる最小仕様に落とす。
数値・ルール・UI要素を具体的に書く。

## 6. 必要な作業
- 実装作業
- 素材作業
- テキスト作業
- サウンド作業
- QA作業

## 7. Issue案
1 Issue = 1 PR になる粒度で、必要なIssue案を作る。
ブランチ名案も添える。

## 8. QA観点
- 面白さ（1回やったらもう1回やりたくなるか）
- テンポ（操作の反応速度・待ち時間）
- 操作性（ボタンサイズ・タップしやすさ）
- スマホ表示（はみ出し・横スクロール・フォントサイズ）
- バグ（数値破壊・進行不能・NaN・undefined）
- 分かりやすさ（初見で何をすればいいか分かるか）

## 9. リファクタリング候補
挙動を変えずに整理できる箇所だけを挙げる。新機能追加とは必ず分ける。
今は触らない方がいい箇所も明記する。

## 10. 今日の完成判定
以下の8条件を今日の差分が満たすかチェックリスト形式で書く。

1. プレイヤーが何をすればいいか分かる
2. プレイヤーが何かを選べる、または操作できる
3. 選択や操作の結果が分かる
4. 成功・失敗・報酬・成長・発見・演出のいずれかがある
5. 次にやりたいことが残る
6. スマホで最低限操作できる
7. バグで進行不能にならない
8. 今日の差分として説明できる

## 11. 最初の1手
今日最初に着手するべき作業を1つだけ、短く提示する。
コマンド例があれば添える。
