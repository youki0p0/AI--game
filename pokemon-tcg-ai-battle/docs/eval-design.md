# ポケモンTCG(CABT) 静的評価値関数 設計書

将棋の静的評価関数(駒得 + 位置/玉安全 の重み付き和)を参考に、PTCG の局面を
自分視点のスカラー値(正=自分有利)へ写像する評価値関数 `state_eval` を設計・実装した。

方針上の位置づけ: 「強いエージェントより先に評価値を作って試合検証する」。
評価値は探索の葉(リーフ)を点数化する静的関数であり、まずこの関数の妥当性を
ランダム対戦の記録で検証してから、評価値を使う greedy エージェントへ進む。

実装: `eval/state_eval.py`
- `state_eval(current: dict, my_index: int, weights=None) -> float`(engine 非依存の純関数)
- 特徴量は `prize_diff` / `material_diff` / `threat` / `resources` に分解
- 重みは `Weights`(dataclass)で名前付き定数化しチューニング可能

---

## 1. 将棋評価値とのアナロジー対応表

| 将棋の概念 | 将棋での意味 | PTCG への翻訳 | 実装上の特徴量 |
|---|---|---|---|
| 駒得(materialの中核) | 盤上の駒の損得 | **サイド差** = 勝利への距離 | `prize_diff`(支配項・最大重み) |
| 駒得(盤上戦力) | 駒の枚数・働き | **盤面マテリアル差** = 在場ポケモンの価値合計差 | `material_diff` |
| 玉の安全度 | 王の堅さ・寄り筋 | **アクティブの HP 余裕** + **状態異常** | `pokemon_value` の HP項, `threat` の状態異常項 |
| 速度・手番(寄せの速さ) | 相手玉に迫る速度 | **KO 可能性**(概算ダメージ vs 相手HP) | `threat` の can_ko / will_be_koed |
| 駒の効率・働き | 遊び駒/働く駒 | **エネルギー充足**(技要求に届くか)・進化段階・ツール | `pokemon_value` の energy/stage/tool項 |
| 持ち駒・手の広さ | 選択肢の多さ | **資源**(手札・山札・ベンチ展開・総エネルギー) | `resources`(小重み) |
| 詰み | 勝敗確定 | **終局**(result != -1) | `state_eval` の terminal 分岐 |

桁感(将棋でいう「歩=100、玉の詰み=∞」に相当):

```
サイド差(±1000/枚) >> マテリアル差(±数百/体) > 脅威(±数十〜250) > 資源(±数〜数十)
終局 = ±100000(他の全項を圧倒する確定値)
```

サイド差を支配項に置くのは、PTCG の勝利条件が「自分のサイドを先に取り切る(=相手ポケモンを
KO してサイドを取る)」であり、盤面マテリアルや状態異常はあくまでサイドを取るための
中間目標だからである。将棋で「駒得しても玉を詰まされたら負け」なのと同様に、
ここでは「マテリアル有利でもサイドを先に取られたら負け」を桁差で表現している。

## 2. 特徴量一覧と重みの初期値・根拠

すべて **自分視点の差分(正=自分有利)**。相手の隠れ情報(手札中身・裏サイドの中身)は
使わず「数」だけを使う。`my_index` で自分/相手を取り違えないよう、相手は常に `1 - my_index`。

### 2.1 支配項: サイド差 `prize_diff`

```
prize_diff = W.prize_diff * (相手の残サイド数 - 自分の残サイド数)
```

- サイド枚数は `players[i]["prize"]`(裏向き None を含む list)の **長さ** で数える。
- 「残サイドが少ない側ほど勝利に近い」。自分の残りが少ない=取った数が多い=有利。
- `W.prize_diff = 1000.0`。1サイド差で 1000 点。6→0 のスイングで最大 ±6000 点になり、
  マテリアル差(典型 ±数百〜千)を確実に上回るよう設計。

| 重み | 初期値 | 根拠 |
|---|---|---|
| `prize_diff` | 1000.0 | 勝利条件直結の支配項。マテリアルより一桁上 |

### 2.2 盤面マテリアル `material_diff`

```
material_diff = W.material_diff * (Σ 自分の場のポケモン価値 - Σ 相手の場のポケモン価値)
```

場のポケモン = `active`(0-1体) + `bench`。1体の価値 `pokemon_value`:

| 要素 | 重み(初期値) | 根拠 |
|---|---|---|
| サイド価値 `w_prize_value` | 300.0 × prize_value | KO されると相手が取るサイド数(megaEx=3, ex=2, 他=1)。盤面に残っている「サイドの塊」の価値。buddy の `prize_count()*1000` を踏襲しつつ prize_diff より控えめ |
| HP 余裕 `w_hp_ratio` | 120.0 × (hp/maxHp) | 玉の体力相当。満タンで満点、瀕死で 0。KO 耐性の連続量 |
| エネルギー充足 `w_energy_ready` | 90.0 × min(have, 最安技要求) | 技を撃てる状態かどうか。技を撃てない戦力は遊び駒 |
| 余剰エネルギー `w_energy_extra` | 15.0 × max(0, have-要求) | 要求を超える分は小さく評価(過剰投資の逓減) |
| 進化段階 `w_stage2/stage1` | 60.0 / 30.0 | これまでの投資量(stage2 は手数2回ぶん) |
| ツール `w_tool` | 25.0 × ツール数 | 付与済みの強化 |

`prize_value` は `engine.cg.api.all_card_data()` の `megaEx`/`ex` を一次情報として参照。
最安技の要求エネルギーは `all_attack()` の各 `energies` の長さの最小値。
カードデータをロードできない環境では prize_value=1、技要求=不明(エネルギーを薄く評価)に
フォールバックする。

> 注: buddy の `prize_count()` はエネルギー(id=12)やツール(id=1172+Lillie)でサイド数を
> 減らす個別カード補正を持つが、これはカード固有知識のため汎用評価には入れていない。
> 必要ならカード ID ホワイトリストで後付けできる。

### 2.3 脅威/テンポ `threat`

アクティブ同士の打ち合いを概算する(弱点/抵抗/効果テキストは無視した粗い見積り)。

| 要素 | 重み(初期値) | 根拠 |
|---|---|---|
| 自分が相手を KO 可能 `w_can_ko` | +250.0 | 概算最大ダメージ ≥ 相手アクティブ HP なら次手で取れる |
| 相手が自分を KO 可能 `w_will_be_koed` | -200.0 | 取られる側のリスク。攻撃側より少し軽く(手番で先に動ける前提) |
| 与ダメ割合 `w_damage_ratio` | ±80.0 × (dmg/HP) | KO に届かなくても削り進捗をテンポとして評価 |

概算ダメージ `_best_attack_damage` = 付与エネルギーで撃てる技のうち最大の素ダメージ。

状態異常(`players[i]` の `poisoned/burned/asleep/paralyzed/confused`):
自分アクティブにあると減点、相手アクティブにあると加点。

| 状態 | 重み(初期値) | 根拠 |
|---|---|---|
| `w_sc_sleep` / `w_sc_paralyze` | 70.0 | 行動不能級。攻撃・逃げを封じる |
| `w_sc_poison` | 40.0 | 毎ターン削れる |
| `w_sc_confuse` | 35.0 | 攻撃失敗リスク |
| `w_sc_burn` | 30.0 | 削り + たまに継続 |

### 2.4 資源 `resources`(重み小)

| 要素 | 重み(初期値) | 根拠 |
|---|---|---|
| 手札差 `w_hand` | 8.0 | 選択肢の広さ。相手は handCount のみ既知 |
| 山札差 `w_deck` | 1.5 | 山切れ負けの遠い回避。重みは最小 |
| ベンチ展開差 `w_bench` | 25.0 | 後続戦力・KO されても立て直せる |
| 総エネルギー差 `w_total_energy` | 6.0 | 盤面に乗ったエネルギー総量 |

### 2.5 終局・無効入力の扱い

- `current is None` → 0.0(デッキ選択フェーズ等)
- `result == 2`(引分) → 0.0
- `result == my_index`(自分勝ち) → +`terminal`(=100000.0)
- `result == 1 - my_index`(相手勝ち) → -`terminal`
- `players` が2要素未満 / `my_index` が 0,1 以外 → 0.0(防御的)

## 3. 検証プロトコル(本題)

評価値関数は「強さ」ではなく「最終結果との整合」で検証する。エージェント実装より先に
評価値の妥当性を測る。ハーネス(jen-builder 製 `play_game`)から `state_eval` を呼べる
純関数設計にしてあるため、対戦ログに評価値を差し込むだけで以下が測れる。

### (a) 相関検証: 評価値の符号は最終結果と一致するか

**手順**

1. ランダム同士(または弱ベースライン同士)の対戦を N 局(例 N=300〜1000)実行。
2. 各手番(各 MAIN 局面)で `state_eval(current, current["yourIndex"])` を記録。
   併せて `turn`、最終的な `result` を保持。
3. 「手番プレイヤー視点の評価値の符号」と「その手番プレイヤーが最終的に勝ったか」の
   一致を集計する。

**指標**

- **符号一致率(sign agreement)**: `sign(eval) == (winner==viewer ? +1 : -1)` の割合。
  引分(result=2)とeval=0は集計から除外、または別カウント。
- **ターン進行 vs 一致率**: 横軸を「ゲーム進行度」(`turn / そのゲームの総ターン`)を
  10ビンに離散化、縦軸を各ビンの符号一致率。
  期待: **終盤ほど一致率が単調に上がる**(序盤は情報が乏しく 50% 近辺、
  サイドが動き始める中盤以降で上昇、終盤で 0.8 以上)。
  これは「静的評価が終局に近いほど正確」という健全性の表れ。
- **AUC 的な見方**: 各局面を (eval_value, 勝敗ラベル) のサンプルとみなし、
  「eval が高い局面ほど勝率が高い」かを ROC-AUC で測る。
  AUC=0.5 は無相関、1.0 は完全分離。全局面プールと、終盤局面のみのプールで分けて算出。
- **キャリブレーション(任意)**: eval をシグモイド `p = 1/(1+exp(-eval/scale))` で
  勝率推定に変換し、推定勝率ビンごとの実勝率と比較(信頼度図)。`scale` は
  サイド差の重み(≈1000)を目安に決める。

**実装メモ**: 集計は対戦ログ(評価値・turn・result)を CSV/JSONL に出すだけで、
評価値関数自体は変更しない。重みチューニングはこの一致率/AUC を目的関数にして回す。

### (b) greedy-eval エージェント

各 MAIN 選択で、**option ごとに結果局面を評価して最善を選ぶ** 1手読みエージェント。

```
def greedy_eval_agent(obs):
    best_i, best_v = None, -inf
    for i, opt in enumerate(obs.select.option):
        next_current = simulate_apply(obs, i)        # 結果局面の取得(後述)
        v = state_eval(next_current, my_index)
        if v > best_v: best_v, best_i = v, i
    return [best_i]
```

**結果局面の取得**: 将来 engine の `search_begin` / `search_step`(`cg/api.py`)を使う。

- `search_begin(observation, your_deck, your_prize, opponent_deck, opponent_prize,
  opponent_hand, opponent_active)` で root 探索状態を作り、
  `search_step(search_id, [option_index])` で各 option を適用した次状態
  (`SearchState.observation.current`)を得て `state_eval` で評価する。
- 隠れ情報(相手のデッキ/サイド/手札の中身)は予測 ID を渡す必要がある。評価値自体は
  数しか使わないが、search_begin は中身の ID を要求するため、最低限の予測
  (例: 既知の自分デッキ構成や全カード中の頻出基本ポケモン)を埋める。
- 探索は MAIN の各 option について1手ぶん展開し、評価後 `search_release` / `search_end` で解放。
- **本タスクでは設計のみ(未実装で可)**。実装は engine 側のハーネス整備後に着手する。

**簡易版(代替案・search 不要)**: 結果局面を作らず、option 種別ヒューリスティック
(buddy 流)で `state_eval` の各特徴量を近似的に増減させて擬似評価する。

- `ATTACK`: 相手アクティブの概算 KO 判定 → KO なら prize_diff/threat を仮想的に加点。
- `EVOLVE` / `ATTACH`: 対象 `pokemon_value` の増分(進化段階・エネルギー充足)を加点。
- `PLAY`(ポケモン): ベンチ展開 +1 ぶんの material/resources 増分。
- `RETREAT`: 状態異常解消・有利なアクティブへの入替なら threat 改善ぶん。
- `END`: 現局面の `state_eval` をそのまま(基準値)。

簡易版は結果局面を厳密に再現しないが、search ハーネスが揃う前のスモーク用に使える。

### (c) 受入基準の素案

| 観点 | 基準(素案) | 測り方 |
|---|---|---|
| 評価-勝者 一致率(終盤) | 進行度 80% 以降のビンで符号一致率 **> 0.80** | (a) のターン進行 vs 一致率 |
| 全局面 AUC | 全局面プールで ROC-AUC **> 0.65**、終盤プールで **> 0.85** | (a) の AUC |
| greedy vs random 勝率 | greedy-eval が random 相手に **勝率 > 60%**(N≥200局) | (b) を対戦ハーネスで回す |
| greedy vs buddy 勝率 | 当面の参考値。**勝率 > 30%** を最初の足切り | 同上(buddy は強めなので低めに設定) |
| 単調性 | 進行度ビンで一致率がおおむね単調増加 | (a) の折れ線 |

数値(X=60%, Y=80% 等)は初期スモークの分布を見て調整する前提のプレースホルダ。

## 4. 設計上の制約・既知の割り切り

- **概算ダメージ**は弱点(×2)・抵抗(-30)・効果テキストを無視している。
  buddy は弱点/抵抗を見ているため、threat 項の精度向上余地がある(後付け可能)。
- **prize_value** は ex/megaEx のみ。buddy のカード固有減算(エネ id=12 等)は未反映。
- **裏向き情報**は数のみ使用(prize は list 長、相手 hand は handCount)。中身は見ない。
- `state_eval` は副作用なしの純関数。`current=None` / 終局 / 空 players を安全に処理。
- カードデータは初回呼び出し時に遅延ロードしてキャッシュ。ロード失敗時もフォールバックで
  有限値を返す(評価精度は落ちるがクラッシュしない)。

## 5. スモーク確認結果

`python3 eval/state_eval.py`(engine/deck.pkl で `battle_start` 初期局面を1つ評価):

```
[smoke] my_index=0
[smoke] state_eval = 0.0
[smoke] breakdown  = {'prize_diff': 0.0, 'material_diff': 0.0, 'threat': 0.0,
                      'resources': 0.0, 'total': 0.0, 'terminal': False}
[smoke] finite float? True
```

初期局面は turn=0 のセットアップ前(active 空・prize 0・hand 0)で完全対称のため
全項 0.0、有限 float を返すことを確認。

人工局面での符号検証(`eval/state_eval.py` の関数を直接呼び出し):

| ケース | 期待 | 実測 total |
|---|---|---|
| 自分サイド残2 vs 相手残6 | 大きく正 | +4978.6(prize_diff +4000 が支配) |
| 同サイド・自分 Mega Lucario満タン vs 相手瀕死 | 正 | +1723.0(material +1350) |
| 上を my_index=1 視点で評価 | 符号反転 | -1673.0 |
| 自分アクティブが眠り(他対称) | 負 | -70.0 |
| 終局 自分勝ち / 負け / 分 | +大 / -大 / 0 | +100000 / -100000 / 0 |
| current=None / 空 players | 0 | 0.0 / 0.0 |

支配項(サイド差)がマテリアル・脅威・資源を桁で上回ること、視点反転で符号が反転すること、
終局・無効入力が安全に処理されることを確認した。
